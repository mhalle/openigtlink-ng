# OpenIGTLink Protocol — v1/v2/v3

**Status:** Authoritative for the v1/v2/v3 wire protocol. The
machine-readable schemas under [`../schemas/`](../schemas/) are
the structured form of the same content; this document carries
the prose explanation, framing rules, and the cross-cutting
behavior (CRC, byte order, transport, ASCII strictness) that
isn't naturally per-message. Where this document and the schemas
agree, both are normative; where they disagree, the
[`../corpus/`](../corpus/) test vectors decide.

A small list of known-pending items (the "v4 / NG" track and a
few open questions on framing exactness) is enumerated under
[Open items](#open-items-todo) at the bottom — those are the
parts genuinely in flight, not the doc as a whole.

**Audience:** implementers of OpenIGTLink-compatible software,
auditors verifying SOUP compliance, and developers of new
language ports.

## Overview

OpenIGTLink is a binary, length-prefixed message protocol for real-time
communication in image-guided therapy and surgical navigation. Messages
are exchanged over TCP, UDP, or WebSocket. Every message consists of:

1. A fixed-size **header** (58 bytes).
2. In v2 and v3, an optional **extended header** of variable length.
3. A **body** whose layout is determined by the message type.
4. In v2 and v3, an optional **metadata block** at the end of the body.

All multi-byte integers and floats on the wire are **big-endian**.

## Protocol versions

The protocol's `version` field in the header (`uint16`) takes values:

- `1` — original format. Header + body only. No metadata, no extended header.
- `2` — adds metadata block at the end of the body.
- `3` — adds the extended header; metadata block position adjusted.

Peers negotiate version implicitly: a receiver accepts any supported
version the sender emits. A `CAPABILITY` message exchange enumerates
supported message types.

**Legacy note:** the v2→v3 transition repositioned the metadata block.
v3 places metadata at the body *end*, after the content, not after the
extended header. This caused several independent implementations to
misplace it; the corpus pins the correct offset.

## Header

58 bytes, fixed layout. All fields big-endian unless noted.

| Offset | Size | Type      | Field         | Notes |
|-------:|-----:|-----------|---------------|-------|
|     0  |    2 | uint16    | version       | 1, 2, or 3 |
|     2  |   12 | char[12]  | type          | ASCII, null-padded message type string |
|    14  |   20 | char[20]  | device_name   | ASCII, null-padded |
|    34  |    8 | uint64    | timestamp     | seconds (upper 32 bits) + fraction (lower 32 bits) since Unix epoch |
|    42  |    8 | uint64    | body_size     | bytes in body (includes extended header and metadata in v2/v3) |
|    50  |    8 | uint64    | crc           | CRC-64 of the body bytes |

**Legacy note:** the `timestamp` is split into two `uint32`s (seconds
and fraction-of-second) on the wire but is often treated as a single
`uint64` in implementations. The fraction field uses the high bit of
the low word for the sign — this should not be relied on.

### CRC

CRC-64 of the entire body (including extended header and metadata
where present). The polynomial is ECMA-182 (the same variant used by
xz/lzma). Implementations MUST verify CRC before acting on any body
content. A CRC mismatch is a hard rejection; implementations MUST NOT
attempt partial parsing.

## Extended header (v3 only)

Variable-length, located at the start of the body in v3 messages.

| Offset | Size | Type   | Field                 | Notes |
|-------:|-----:|--------|-----------------------|-------|
|     0  |    2 | uint16 | ext_header_size       | bytes in this extended header |
|     2  |    2 | uint16 | metadata_header_size  | bytes in the metadata-index section |
|     4  |    4 | uint32 | metadata_size         | bytes in the metadata-body section |
|     8  |    4 | uint32 | message_id            | opaque sender-assigned identifier |
|   12+  |      | bytes  | (extension fields)    | reserved for future use |

The actual extended header size is `ext_header_size`, which may
exceed 12 bytes in future revisions. Implementations MUST skip any
bytes beyond the known fields up to `ext_header_size` without error.

## Metadata block (v2 and v3)

A metadata block carries arbitrary key/value pairs. Layout:

- **v2:** appended after content; located by subtracting content size from body_size.
- **v3:** located by the sizes declared in the extended header.

### Metadata index (header section)

- `uint16 count` — number of entries
- Per entry, in index order:
  - `uint16 key_size`
  - `uint16 value_encoding` — IANA MIBenum charset code (3 = US-ASCII; 106 = UTF-8)
  - `uint32 value_size`

### Metadata body

Sequential, packed, no padding:

- key bytes (`key_size` bytes, UTF-8, no terminator)
- value bytes (`value_size` bytes, encoded per `value_encoding`)

### Validation requirements

- Key size MUST be > 0.
- Total size of all entries MUST equal the declared `metadata_size`.
- Key and value size sums MUST NOT overflow the declared metadata region.
- Keys SHOULD be unique; behavior on duplicate keys is implementation-defined.

**Legacy note:** several audit findings in the existing library's
metadata parser stem from missing integer-overflow checks on these
size fields. A conformant implementation rejects metadata blocks whose
declared sizes do not fit within the body.

## Body

The body layout is fully determined by the `type` field. See the
machine-readable schemas under [`../schemas/`](../schemas/) for the
normative description of each message type. The prose description
below is a human-readable overview.

### Message type categories

- **Data messages** — TRANSFORM, POSITION, IMAGE, NDARRAY, POINT,
  TRAJECTORY, TDATA, QTDATA, STRING, SENSOR, POLYDATA, COLORTABLE,
  IMGMETA, LBMETA, VIDEOMETA, VIDEO, STATUS, CAPABILITY, BIND, UNIT,
  COMMAND.
- **Query messages** — `GET_*` (request a current snapshot of a data
  type), `STT_*` (request a stream), `STP_*` (stop a stream),
  `RTS_*` (reply-to-start / reply-to-stop status). See the
  [Query/control type_id reference](#querycontrol-typeid-reference)
  section below for the complete mapping.

### Message type strings

Each type has a 12-byte ASCII string, null-padded. The canonical
string is specified in the schema for each message type. Type strings
are case-sensitive.

**Legacy note:** the 12-byte width has caused several historical
audit issues when type strings were memcpy'd without null
termination. Implementations MUST treat the field as fixed-width
bytes, not a C string, and MUST null-terminate before any use as a
C string.

### STATUS

Reports the operational status of a device or the outcome of a
previous request. See [`../schemas/status.json`](../schemas/status.json)
for the normative wire layout.

**Body layout** (variable-size, minimum 31 bytes):

| Offset | Size | Type      | Field          |
|-------:|-----:|-----------|----------------|
|     0  |    2 | uint16    | code           |
|     2  |    8 | int64     | subcode        |
|    10  |   20 | char[20]  | error_name     |
|    30  |  N+1 | char[N+1] | status_message |

`status_message` occupies all remaining body bytes. Its final byte
MUST be `0x00`; the string value is the preceding `N` bytes. An empty
status_message is represented as a single `0x00` byte (N=0), giving a
minimum body size of 31 bytes.

**Status codes** (defined values):

| Value | Name              | Meaning |
|------:|-------------------|---------|
|     0 | INVALID           | Sentinel, not a valid runtime status |
|     1 | OK                | Normal operation |
|     2 | UNKNOWN_ERROR     | Generic error |
|     3 | PANIC_MODE        | Emergency (upstream constant spelled `PANICK_MODE`) |
|     4 | NOT_FOUND         | File, configuration, or device not found |
|     5 | ACCESS_DENIED     | |
|     6 | BUSY              | |
|     7 | TIME_OUT          | Timeout or connection lost |
|     8 | OVERFLOW          | Overflow or cannot be reached |
|     9 | CHECKSUM_ERROR    | |
|    10 | CONFIG_ERROR      | |
|    11 | RESOURCE_ERROR    | Insufficient memory, storage, etc. |
|    12 | ILLEGAL_INSTRUCTION | Unknown or unsupported instruction |
|    13 | NOT_READY         | Device starting up |
|    14 | MANUAL_MODE       | Device does not accept commands |
|    15 | DISABLED          | |
|    16 | NOT_PRESENT       | |
|    17 | UNKNOWN_VERSION   | |
|    18 | HARDWARE_FAILURE  | |
|    19 | SHUT_DOWN         | Exiting or shutdown in progress |

Receivers MUST accept unknown code values without rejection, so the
enumeration can grow over time without breaking compatibility.

### STRING

Carries a character string with an explicit character-encoding hint.
See [`../schemas/string.json`](../schemas/string.json) for the normative
wire layout.

**Body layout** (variable-size, minimum 4 bytes):

| Offset | Size | Type    | Field    |
|-------:|-----:|---------|----------|
|     0  |    2 | uint16  | encoding |
|     2  |    2 | uint16  | length   |
|     4  |    N | byte[N] | value    |

`value` is `length` bytes of string data in the encoding specified by
the `encoding` field. There is no terminator byte. An empty string is
legal (`length=0`, body size 4).

The `encoding` field is an IANA MIBenum value. Common values:
- `3` — US-ASCII
- `106` — UTF-8

Receivers MUST accept unknown encoding values without rejection; if
the encoding is not recognized, the payload MAY be treated as opaque
bytes. The 16-bit `length` field caps the maximum string size at
65535 bytes.

### SENSOR

Reports a vector of scalar sensor readings with an attached SI unit
descriptor. See [`../schemas/sensor.json`](../schemas/sensor.json) for
the normative wire layout.

**Body layout** (variable-size; total = 10 + `larray` × 8 bytes):

| Offset | Size | Type        | Field   |
|-------:|-----:|-------------|---------|
|     0  |    1 | uint8       | larray  |
|     1  |    1 | uint8       | status  |
|     2  |    8 | uint64      | unit    |
|    10  |  N×8 | float64[N]  | data    |

Where `N = larray` (range 0..255). Maximum body size is therefore
10 + 255 × 8 = 2050 bytes.

The `unit` field is a bit-packed descriptor of an SI unit (prefix
code + up to six base- or derived-unit codes with signed exponents).
At the wire level it is a single big-endian `uint64`; decoding its
internal structure is optional and can be deferred to a helper.

The `status` byte has no standardized values. Receivers SHOULD ignore
unrecognized values and MUST NOT reject a message based on its
content.

### CAPABILITY

Advertises the list of OpenIGTLink message types a device accepts.
See [`../schemas/capability.json`](../schemas/capability.json) for the
normative wire layout.

**Body layout** (variable-size; always a multiple of 12 bytes):

| Offset | Size  | Type            | Field            |
|-------:|------:|-----------------|------------------|
|     0  | 12×N  | char[12]×N      | supported_types  |

Each element is an OpenIGTLink message-type identifier as a 12-byte
ASCII string, null-padded on the right. The number of entries is
implicit: `N = body_size / 12`.

A body whose size is not a multiple of 12 is malformed and MUST be
rejected. An empty list (`N = 0`, `body_size = 0`) is legal but
unusual.

**Typical usage:** sent in response to a `GET_CAPABIL` query during
session setup. Peers inspect the list to decide which message types
to send on the session.

### TDATA

Stream of tracked-tool poses using a 3×4 matrix per tool. See
[`../schemas/tdata.json`](../schemas/tdata.json).

**Element layout** (70 bytes per element; body = 70 × N):

| Offset | Size | Type         | Field      |
|-------:|-----:|--------------|------------|
|     0  |   20 | char[20]     | name       |
|    20  |    1 | uint8        | type       |
|    21  |    1 | uint8        | reserved   |
|    22  |   48 | float32[12]  | transform  |

`transform` is byte-identical to a TRANSFORM message body —
column-major 3×4, wire order R11, R21, R31, R12, …, TX, TY, TZ.
`type`: 1 = tracker reference, 2 = 6D, 3 = 3D, 4 = 5D. `N =
body_size / 70`; a non-multiple body is malformed.

### QTDATA

Stream of tracked-tool poses using position + quaternion per tool.
See [`../schemas/qtdata.json`](../schemas/qtdata.json).

**Element layout** (50 bytes per element; body = 50 × N):

| Offset | Size | Type         | Field      |
|-------:|-----:|--------------|------------|
|     0  |   20 | char[20]     | name       |
|    20  |    1 | uint8        | type       |
|    21  |    1 | uint8        | reserved   |
|    22  |   12 | float32[3]   | position   |
|    34  |   16 | float32[4]   | quaternion |

Quaternion order is (qx, qy, qz, w). `type` values as in TDATA.
`N = body_size / 50`; a non-multiple body is malformed.

### POINT

List of annotated 3D points (landmarks, fiducials, targets). See
[`../schemas/point.json`](../schemas/point.json).

**Element layout** (136 bytes per element; body = 136 × N):

| Offset | Size | Type         | Field       |
|-------:|-----:|--------------|-------------|
|     0  |   64 | char[64]     | name        |
|    64  |   32 | char[32]     | group_name  |
|    96  |    4 | uint8[4]     | rgba        |
|   100  |   12 | float32[3]   | position    |
|   112  |    4 | float32      | radius      |
|   116  |   20 | char[20]     | owner       |

`N = body_size / 136`; a non-multiple body is malformed.

### TRAJECTORY

List of 3D trajectories (entry → target paths). See
[`../schemas/trajectory.json`](../schemas/trajectory.json).

**Element layout** (150 bytes per element; body = 150 × N):

| Offset | Size | Type         | Field       |
|-------:|-----:|--------------|-------------|
|     0  |   64 | char[64]     | name        |
|    64  |   32 | char[32]     | group_name  |
|    96  |    1 | int8         | type        |
|    97  |    1 | int8         | reserved    |
|    98  |    4 | uint8[4]     | rgba        |
|   102  |   12 | float32[3]   | entry_pos   |
|   114  |   12 | float32[3]   | target_pos  |
|   126  |    4 | float32      | radius      |
|   130  |   20 | char[20]     | owner_name  |

`type`: 1 = entry-only, 2 = target-only, 3 = entry-and-target.
Both positions occupy wire bytes regardless of `type`; the unused
one is conventionally zeros. `N = body_size / 150`.

### IMGMETA

Directory of IMAGE volumes available on a server. See
[`../schemas/imgmeta.json`](../schemas/imgmeta.json).

**Element layout** (260 bytes per element; body = 260 × N):

| Offset | Size | Type         | Field         |
|-------:|-----:|--------------|---------------|
|     0  |   64 | char[64]     | name          |
|    64  |   20 | char[20]     | device_name   |
|    84  |   32 | char[32]     | modality      |
|   116  |   64 | char[64]     | patient_name  |
|   180  |   64 | char[64]     | patient_id    |
|   244  |    8 | uint64       | timestamp     |
|   252  |    6 | uint16[3]    | size          |
|   258  |    1 | uint8        | scalar_type   |
|   259  |    1 | uint8        | reserved      |

`patient_name` and `patient_id` are PHI in most clinical
deployments. `scalar_type` matches IMAGE's scalar-type codes.
`N = body_size / 260`.

### LBMETA

Directory of label-map regions. See
[`../schemas/lbmeta.json`](../schemas/lbmeta.json).

**Element layout** (116 bytes per element; body = 116 × N):

| Offset | Size | Type         | Field        |
|-------:|-----:|--------------|--------------|
|     0  |   64 | char[64]     | name         |
|    64  |   20 | char[20]     | device_name  |
|    84  |    1 | uint8        | label        |
|    85  |    1 | uint8        | reserved     |
|    86  |    4 | uint8[4]     | rgba         |
|    90  |    6 | uint16[3]    | size         |
|    96  |   20 | char[20]     | owner        |

Multiple LBMETA elements may share `device_name` with different
`label` values, advertising a multi-label IMAGE. `N = body_size / 116`.

### VIDEOMETA

Directory of available video sources. See
[`../schemas/videometa.json`](../schemas/videometa.json). Introduced
in v3.1 (July 2017).

**Element layout** (322 bytes per element; body = 322 × N):

| Offset | Size | Type         | Field         |
|-------:|-----:|--------------|---------------|
|     0  |   64 | char[64]     | name          |
|    64  |   64 | char[64]     | device_name   |
|   128  |   64 | char[64]     | patient_name  |
|   192  |   64 | char[64]     | patient_id    |
|   256  |    2 | int16        | zoom_level    |
|   258  |    8 | float64      | focal_length  |
|   266  |    6 | uint16[3]    | size          |
|   272  |   48 | float32[12]  | matrix        |
|   320  |    1 | uint8        | scalar_type   |
|   321  |    1 | uint8        | reserved      |

The `device_name` inside the element is 64 bytes — wider than the
20-byte device_name suffix used by IMGMETA, LBMETA, POINT, and
TRAJECTORY. Clients synthesizing a GET_VIDEO request from this field
MUST truncate or reject values that do not fit the 20-byte
fixed-header device-name width.

The `matrix` field carries four float32[3] groups in row order:
(TX,TY,TZ) = norm_i × pixel_size_i, (SX,SY,SZ) = norm_j ×
pixel_size_j, (NX,NY,NZ) = norm_k × pixel_size_k, (PX,PY,PZ) = frame
origin in millimeters. Same convention as IMAGE's orientation matrix.

`N = body_size / 322`.

### IMAGE

Pixel data for a 2D frame or 3D volume, with in-band orientation and
optional subvolume windowing. See
[`../schemas/image.json`](../schemas/image.json).

**Header layout** (fixed 72 bytes, followed by the pixel byte array):

| Offset | Size | Type         | Field           |
|-------:|-----:|--------------|-----------------|
|     0  |    2 | uint16       | header_version  |
|     2  |    1 | uint8        | num_components  |
|     3  |    1 | uint8        | scalar_type     |
|     4  |    1 | uint8        | endian          |
|     5  |    1 | uint8        | coord           |
|     6  |    6 | uint16[3]    | size            |
|    12  |   48 | float32[12]  | matrix          |
|    60  |    6 | uint16[3]    | subvol_offset   |
|    66  |    6 | uint16[3]    | subvol_size     |
|    72  |  var | uint8[]      | pixels          |

`header_version` is the IMAGE message format version (currently `1`),
not the outer protocol version. Receivers MUST reject
`header_version` values they do not implement.

`scalar_type` codes match IMGMETA / VIDEOMETA: 2=int8, 3=uint8,
4=int16, 5=uint16, 6=int32, 7=uint32, 10=float32, 11=float64.

`endian` applies to the pixel bytes only (1=big, 2=little); the
72-byte header is always big-endian / network order.

`coord` = 1 (RAS) or 2 (LPS). Interpretation of the orientation axes;
not a pixel-buffer transform.

`matrix` is four float32[3] groups in row order: norm_i·pixel_size_i,
norm_j·pixel_size_j, norm_k·pixel_size_k, origin (mm).

`subvol_offset + subvol_size <= size` componentwise — enforce this
before allocating; it is unchecked in the upstream library.

**Pixel byte count** (expected `body_size - 72`):

```
pixel_bytes = num_components
            * subvol_size[0] * subvol_size[1] * subvol_size[2]
            * bytes_per_scalar(scalar_type)
```

Computation MUST be done in 64-bit arithmetic. Receivers MUST
verify that `body_size` exactly equals `72 + pixel_bytes`; excess or
shortfall is a malformed message. A conformant implementation MUST
also impose an implementation-defined ceiling on `pixel_bytes` before
allocating the receive buffer.

There is no separate IMAGE2 message type; the file
`Testing/img/igtlTestImage2.raw` in the upstream tree is simply the
second test image payload used by the test suite.

### COMMAND

Structured command channel — typically XML, carried with an ID so
replies can correlate. See
[`../schemas/command.json`](../schemas/command.json). Introduced in
v3.0 (January 2017).

**Header layout** (fixed 138 bytes, followed by `length` bytes of
command payload):

| Offset | Size | Type       | Field         |
|-------:|-----:|------------|---------------|
|     0  |    4 | uint32     | command_id    |
|     4  |  128 | char[128]  | command_name  |
|   132  |    2 | uint16     | encoding      |
|   134  |    4 | uint32     | length        |
|   138  |  var | uint8[]    | command       |

`body_size = 138 + length`. `encoding` is an IANA MIBenum value —
typical values are 3 (US-ASCII, the default), 4 (ISO-8859-1), 106
(UTF-8). Receivers MUST validate encoding against the IANA table and
MUST verify `body_size >= 138 + length` before any access to the
command region.

`RTS_COMMAND` replies reuse the COMMAND body layout; the
`command_name` field carries an error string when used in that
direction.

### POSITION

3D position with optional orientation quaternion. See
[`../schemas/position.json`](../schemas/position.json).

**Body layout** (12, 24, or 28 bytes — size-discriminated):

| Offset | Size | Type       | Field       | When present    |
|-------:|-----:|------------|-------------|-----------------|
|     0  |   12 | float32[3] | position    | always          |
|    12  | 0/12/16 | float32[0/3/4] | quaternion | body ≥ 24 |

- **12 bytes** — position only; no orientation.
- **24 bytes** — position + 3-element compressed quaternion
  (qx, qy, qz); receiver reconstructs qw = √(1 − qx² − qy² − qz²).
- **28 bytes** — position + full quaternion (qx, qy, qz, qw).

Any other body_size is malformed and MUST be rejected.

### QTRANS

Position + full quaternion in 28 fixed bytes. See
[`../schemas/qtrans.json`](../schemas/qtrans.json). Introduced in
v3.0 per `Documents/Protocol/qtransform.md`.

**Body layout** (fixed 28 bytes):

| Offset | Size | Type       | Field      |
|-------:|-----:|------------|------------|
|     0  |   12 | float32[3] | position   |
|    12  |   16 | float32[4] | quaternion |

Byte-for-byte identical to POSITION's 28-byte variant, but carried
under a distinct type_id so receivers can dispatch without inspecting
body_size. The upstream C++ library does NOT provide a dedicated
class for QTRANS (users emit POSITION with 28-byte body instead);
spec-compliant receivers SHOULD accept both and treat them as
equivalent.

### NDARRAY

Variable-rank N-dimensional array. See
[`../schemas/ndarray.json`](../schemas/ndarray.json).

**Body layout** (variable):

| Offset | Size   | Type      | Field        |
|-------:|-------:|-----------|--------------|
|     0  |      1 | uint8     | scalar_type  |
|     1  |      1 | uint8     | dim          |
|     2  | dim×2  | uint16[]  | size         |
| 2+dim×2| var    | uint8[]   | data         |

`scalar_type` codes: 2=int8, 3=uint8, 4=int16, 5=uint16, 6=int32,
7=uint32, 10=float32, 11=float64, **13=complex** (16 bytes: two
float64). `data` contains product(size) × bytes_per_scalar bytes in
row-major order. Receivers MUST compute the product in 64-bit
arithmetic and reject on overflow.

### COLORTABLE

Color lookup table for label-map images. See
[`../schemas/colortable.json`](../schemas/colortable.json). Wire
type_id is `COLORT`.

**Body layout** (variable, bounded ≤ 192 KiB):

| Offset | Size | Type    | Field       |
|-------:|-----:|---------|-------------|
|     0  |    1 | int8    | index_type  |
|     1  |    1 | int8    | map_type    |
|     2  |  var | uint8[] | table       |

`index_type`: 3 → 256 entries (uint8 index), 5 → 65536 entries
(uint16 index). `map_type`: 3 → 1 byte/entry, 5 → 2 bytes/entry,
19 → 3 bytes/entry (RGB). Table byte count = N × S. Maximum
body_size = 2 + 65536 × 3 = 196610 bytes.

### VIDEO

Codec-compressed video frame with in-band orientation. See
[`../schemas/video.json`](../schemas/video.json). Introduced in v3.

**Header layout** (fixed 76 bytes + compressed frame payload):

| Offset | Size | Type         | Field          |
|-------:|-----:|--------------|----------------|
|     0  |    2 | uint16       | header_version |
|     2  |    1 | uint8        | endian         |
|     3  |    4 | char[4]      | codec          |
|     7  |    2 | uint16       | frame_type     |
|     9  |    1 | uint8        | coord          |
|    10  |    6 | uint16[3]    | size           |
|    16  |   48 | float32[12]  | matrix         |
|    64  |    6 | uint16[3]    | subvol_offset  |
|    70  |    6 | uint16[3]    | subvol_size    |
|    76  |  var | uint8[]      | frame_data     |

`codec` is a 4-byte FourCC (all 4 bytes significant): `I420`
(uncompressed YUV), `H264`, `VP90` (VP9), `X265`, `O265`
(OpenHEVC), `AV10` (AV1). `frame_data` length = `body_size - 76`;
it is NOT derivable from `subvol_size` because the payload is
compressed.

### BIND

Multi-message container. See
[`../schemas/bind.json`](../schemas/bind.json).

**Body layout** (variable):

| Section | Content |
|---|---|
| ncmessages | uint16 — child count N |
| header_entries | N × 20B struct: type_id char[12] + body_size uint64 |
| nametable_size | uint16 — byte length of name table (must be even) |
| name_table | nametable_size bytes — N null-terminated device names |
| bodies | remaining bytes — N child payloads, each body_size[i] bytes + padding if odd |

The header table declares each child's type and body size; the bodies
section is partitioned using those sizes. Child bodies are raw
payloads (no per-child OpenIGTLink header). A receiver MUST validate
`body_size` against the sum of all sections before iterating.

### POLYDATA

Polygonal mesh (VTK vtkPolyData). See
[`../schemas/polydata.json`](../schemas/polydata.json).

**Fixed header** (40 bytes = 10 × uint32):

| Offset | Field | Offset | Field |
|-------:|-------|-------:|-------|
|     0  | npoints | 20 | npolygons |
|     4  | nvertices | 24 | size_polygons |
|     8  | size_vertices | 28 | ntriangle_strips |
|    12  | nlines | 32 | size_triangle_strips |
|    16  | size_lines | 36 | nattributes |

**Data sections** (in order after header):

| Section | Size | Content |
|---|---|---|
| points | npoints × 12B | struct(x,y,z float32)[npoints] |
| vertices | size_vertices B | uint32 VTK connectivity |
| lines | size_lines B | uint32 VTK connectivity |
| polygons | size_polygons B | uint32 VTK connectivity |
| triangle_strips | size_triangle_strips B | uint32 VTK connectivity |
| attribute_headers | nattributes × 6B | struct(type u8, ncomponents u8, n u32) |
| attribute_data | remaining B | name table + float32 data arrays |

Topology sections use VTK connectivity encoding: runs of
(N, i1, i2, …, iN). All topology byte sizes MUST be multiples of 4.
Attribute names are null-terminated strings packed with 2-byte
alignment; data arrays are float32 regardless of attribute type.

## Query/control type_id reference

Every data-carrying message type has up to four query/control
companions: `GET_*` (request snapshot), `STT_*` (start streaming),
`STP_*` (stop streaming), `RTS_*` (return status). Most have
trivial bodies; the table below lists every type_id with its body
pattern. Each has a corresponding schema under `spec/schemas/`.

### Body patterns

- **empty** — body_size = 0, no fields.
- **rts_status** — body_size = 1, single `int8 status` (0=OK,
  1=error).
- **stt_tracking** — body_size = 36, `int32 resolution` (ms) +
  `char[32] coord_name`. Used by STT_TDATA and STT_QTDATA.
- **stt_video** — body_size = 8, `char[4] codec` (FourCC) +
  `uint32 time_interval` (ms).
- **get_bind** — variable, selective request body (see BIND).
- **rts_command** — variable, echoes the COMMAND body layout.

### Complete type_id table

|---|---|---|---|---|
| TRANSFORM | GET_TRANS (empty) | STT_TRANS (empty) | STP_TRANS (empty) | RTS_TRANS (rts_status) |

### Type_id truncation policy

Type_ids are at most 12 ASCII bytes (null-padded). When `PREFIX_ + TYPE`
fits in 12 bytes, the full form is used (`GET_POSITION`,
`STT_POLYDATA`). When it does not fit, upstream uses an ad-hoc
abbreviation; implementations SHOULD honor the upstream form for
wire compatibility:

| Full name | Upstream wire form |
|---|---|
| `GET_TRANSFORM` (13 chars) | `GET_TRANS` |
| `STT_TRANSFORM` | `STT_TRANS` |
| `STP_TRANSFORM` | `STP_TRANS` |
| `RTS_TRANSFORM` | `RTS_TRANS` |
| `GET_TRAJECTORY` (14 chars) | `GET_TRAJ` |
| `GET_CAPABILITY` (14 chars) | `GET_CAPABIL` (11 chars) |
| `RTS_CAPABILITY` | `RTS_CAPABIL` |
| `GET_VIDEOMETA` (13 chars) | `GET_VMETA` |

Note: `Documents/Protocol/query.md` in the upstream spec uses
`GET_TRANSFOR` (truncated to 12 chars with trailing `R`) rather than
the factory's `GET_TRANS`. This is a documented spec-vs-code
inconsistency. Implementations MUST emit the factory form
(`GET_TRANS`) to remain interoperable with deployed upstream
clients; parsers MAY additionally accept `GET_TRANSFOR` for spec
compliance.

## Transport

### TCP

The default transport. Messages are sent back-to-back with no framing
other than the header-declared `body_size`. Connection-oriented; no
retry semantics at the protocol level.

### UDP

Used for VIDEO and high-rate tracking streams. Messages that exceed
the path MTU are fragmented using an OpenIGTLink-specific RTP-lookalike
wrapper. The wrapper is not RFC 3550 compliant.

**Legacy note:** the current UDP/RTP wrapper invents its own fragment
markers (`0x8000` begin, `0xE000` end, `0x0000` no-fragment). It is
NOT interoperable with standard RTP implementations. NG (v4) replaces
this with RFC-compliant RTP payload formats; v3 peers continue to use
the legacy wrapper for backward compatibility.

### WebSocket

Each OpenIGTLink message is sent as a single binary WebSocket frame.

### TLS

Not specified by v3. Implementations MAY wrap any transport in TLS
externally. NG defines explicit TLS negotiation and requirements.

## Conformance

An implementation is conformant iff it passes every entry in
[`../corpus/`](../corpus/):

- Decoder test: given `<entry>.bin`, produce a semantic result matching
  `<entry>.json`.
- Encoder test: given `<entry>.json`, produce bytes exactly matching
  `<entry>.bin`.
- Negative test: given an entry under `corpus/negative/`, reject it
  with an error matching the entry's declared expected behavior.

In-library tests are insufficient to establish conformance; the corpus
is the sole source of truth for wire compatibility.

## Known upstream correctness issues

Behavior observed in the reference C++ implementation at pinned SHA
`94244fe` that differs from what a correct implementation should do.
Each item below identifies a bug to be preserved as a *negative-corpus*
expectation (i.e., the corpus will pin the *correct* rejection
behavior, not the buggy upstream behavior). When the hardening commits
from the patched library land upstream, these entries should be
revisited and either resolved or retained as documentation of the
fixed defect.

### U-1: `SetErrorName` strncpy can leave `error_name` non-null-terminated

**Location:** `Source/igtlStatusMessage.cxx:72-77`, `UnpackContent` at the same file `:132-133`.

**Behavior:** The code writes `m_ErrorName[19] = '\0'` and then calls
`strncpy(m_ErrorName, name, 20)`. If `name` is 20 or more bytes,
strncpy writes 20 characters and overwrites the pre-placed null.
Subsequent use of `m_ErrorName` as a C string OOB-reads past the
buffer.

**Required conformant behavior:** either reserve byte 19 as a hard
null (effective 19-character limit) or treat the field as fixed 20
bytes with no null-termination assumption. Receivers MUST defensively
null-terminate before C-string use.

**Maps to audit commit:** upstream hardening branch commit `aade268`
("null termination, inet_addr, type safety").

### U-2: `StatusMessage::UnpackContent` reads before `m_StatusMessage` on short body

**Location:** `Source/igtlStatusMessage.cxx:137`.

**Behavior:** The null-terminator check is
`m_StatusMessage[CalculateReceiveContentSize(isUnpacked) - IGTL_STATUS_HEADER_SIZE - 1]`.
If the attacker sets `body_size` such that the content size is ≤ 30,
this indexes with a negative offset and reads from memory before the
`m_StatusMessage` buffer.

**Required conformant behavior:** reject any STATUS message with
`body_size < 31` before any body access. The minimum body is the
30-byte fixed header plus the single mandatory null terminator byte.

### U-3: `StatusMessage::UnpackContent` casts `m_Content` without size check

**Location:** `Source/igtlStatusMessage.cxx:127-130`.

**Behavior:** The function casts `m_Content` to `igtl_status_header*`
and reads 30 bytes without first verifying `body_size >= 30`. A
malformed short body results in OOB reads or potential null deref.

**Required conformant behavior:** validate body size against the
fixed-header size before any field access.

### U-4: UDP/RTP wrapper fragment markers are not RFC 3550

**Location:** `Source/igtlMessageRTPWrapper.cxx`.

**Behavior:** The UDP fragmentation scheme invents its own markers
(`0x8000` begin, `0xE000` end, `0x0000` no-fragment) rather than using
RFC 6184 / 7798 / 7741 RTP payload formats. Not a correctness bug per
se, but an interop limitation that has also produced a heap overflow
in the fragment reassembler.

**Required conformant behavior:** v3 peers preserve the legacy wrapper
format for wire compatibility. NG (v4) replaces it with RFC-compliant
RTP. See the "Transport: UDP" section above.

### U-5: `StringMessage::UnpackContent` OOB read on attacker-controlled length

**Location:** `Source/igtlStringMessage.cxx:135`.

**Behavior:** The unpacker calls
`m_String.append(string, string_header->length)` without verifying
`length` against the actual body size available. An attacker sending
`body_size=4` with `length=65535` causes a 65535-byte OOB read starting
at offset 4 of the message content buffer.

**Required conformant behavior:** reject any STRING message where
`body_size < 4 + length`. The check MUST occur before any read of the
string payload.

**Maps to audit commit:** upstream hardening branch — STRING is
covered by the general `CalculateReceiveContentSize(isUnpacked)`
pattern introduced in commits `f3640aa`, `5c0499e`, and `4ad0923`
("Fix StringMessage validation order").

### U-6: `SensorMessage::UnpackContent` OOB on attacker-controlled `larray`

**Location:** `Source/igtlSensorMessage.cxx:168`; underlying byte-swap
loop at `Source/igtlutil/igtl_sensor.c:44-47`.

**Behavior:** `UnpackContent` calls `igtl_sensor_convert_byte_order`
before verifying the body is large enough. That function iterates
`i < larray` and byte-swaps `tmp[i]` (8 bytes each) in place. An
attacker sending `body_size=10` (fixed header only) with `larray=255`
causes a 2040-byte OOB read and 2040-byte OOB write immediately past
the receive buffer. The subsequent `data[i]` reads at lines 176–179
amplify the OOB read.

**Required conformant behavior:** reject any SENSOR message where
`body_size < 10 + larray * 8`. Check MUST occur before any data-array
access, including the byte-swap pass.

**Maps to audit commit:** covered by the general
`CalculateReceiveContentSize(isUnpacked)` pattern used across the
hardening branch; SENSOR-specific coverage is part of the
`0ad4b4d` group ("Reject malformed element-based messages with
partial elements").

### U-7: `igtl_image_get_data_size` ignores `num_components` and float scalar types

**Location:** `Source/igtlutil/igtl_image.c:21-54` (size function) and
`Source/igtlutil/igtl_image.c:209-258` (CRC function, which duplicates
the same switch in-line).

**Behavior:** The payload-size computation is

```c
switch (scalar_type) {
  case INT8:  case UINT8:  sp = 1; break;
  case INT16: case UINT16: sp = 2; break;
  case INT32: case UINT32: sp = 4; break;
  default:                 sp = 0; break;
}
data_size = si * sj * sk * sp;
```

Two bugs:

1. `num_components` is not multiplied in. A 3-component RGB or vector
   image reports data_size = 1/3 of its actual wire footprint; the
   sender is willing to emit the full buffer (`ImageMessage::PackContent`
   allocates via `GetSubVolumeImageSize` which does include
   num_components), but the C utility disagrees. A receiver that trusts
   `igtl_image_get_data_size` reads only the first component's worth of
   pixels; the remaining 2/3 of the body appears as trailing garbage or
   is treated as the next message.

2. Scalar types `FLOAT32` (10) and `FLOAT64` (11) fall through to
   `default: sp = 0`. Any float-typed image has a reported data_size of
   0. `igtl_image_get_crc` computes CRC over 0 pixel bytes for such
   images; a receiver verifying CRC against this oracle passes any
   body that merely preserves the 72-byte header intact, regardless of
   pixel content.

**Required conformant behavior:** use a scalar-size table that covers
all eight defined scalar types (matching `ScalarSizeTable` in
`igtlImageMessage.cxx`), and multiply by `num_components`:

```
pixel_bytes = num_components
            * subvol_size[0] * subvol_size[1] * subvol_size[2]
            * scalar_size_table[scalar_type]
```

Reject messages whose scalar_type is not in the table. Compute in
64-bit arithmetic.

**Maps to audit commit:** not yet addressed upstream. Flagged for the
NG reimplementation.

### U-8: `ImageMessage::UnpackContent` performs no body-size validation

**Location:** `Source/igtlImageMessage.cxx:411-472`.

**Behavior:** UnpackContent casts `m_Content` to `igtl_image_header*`
and reads 72 bytes (the entire fixed header) before any size check.
A body smaller than 72 bytes triggers a 72-(body_size) byte OOB read
in `igtl_image_convert_byte_order`. After the header is parsed,
`m_Image = &m_ImageHeader[IGTL_IMAGE_HEADER_SIZE]` is exposed to the
caller with no verification that the remaining bytes cover the
header-declared pixel volume; a caller reading through `m_Image`
into a claimed 1 GiB subvol off a 72-byte body dereferences unmapped
memory.

Secondary concern: `ImageMessage::GetImageSize()` (header line 240)
returns `int` despite the method's own TODO noting the result should
be 64-bit, and `GetSubVolumeImageSize()` (line 255) computes its
product in signed int before returning igtlUint64. Either can
overflow into UB for large volumes even when the per-field values are
individually bounded by uint16.

**Required conformant behavior:** reject any IMAGE message where
`body_size < 72`. After parsing the header, compute
`expected_body = 72 + pixel_bytes` (using the U-7 formula, in 64-bit
arithmetic) and reject unless `body_size == expected_body`. Impose an
implementation-defined ceiling on `pixel_bytes` before allocating.

**Maps to audit commit:** not yet addressed upstream. Flagged for the
NG reimplementation.

### U-9: `CommandMessage::UnpackContent` OOB read on attacker-controlled length

**Location:** `Source/igtlCommandMessage.cxx:193-217`.

**Behavior:** Same family as U-5 (STRING) but with a uint32 length
instead of uint16. UnpackContent casts `m_Content` to
`igtl_command_header*` and reads 138 bytes before any body-size
check. It then appends the command region to an `std::string` as

```c++
this->m_Command.append(command, command_header->length);
```

where `length` is attacker-controlled and may span up to 4 GiB. A
message with `body_size = 138` (header only) and `length = UINT32_MAX`
causes a 4 GiB OOB read starting at offset 138 of the receive
buffer — a 65536× larger window than the STRING finding.

**Required conformant behavior:** reject any COMMAND message where
`body_size < 138` or `body_size < 138 + length`, before any access
to the command region. The two checks MUST occur before the
byte-swap pass that decodes `length` from network order; use a
peek-and-swap pattern that reads into a local copy.

**Maps to audit commit:** not yet addressed upstream. Flagged for
the NG reimplementation.

### U-10: `CommandMessage::SetCommandName` off-by-one via strcpy

**Location:** `Source/igtlCommandMessage.cxx:73-82` (and:222-229 for
`RTSCommandMessage::SetCommandErrorString`, :103-112 for the
`std::string` overloads — same bug in four places).

**Behavior:** The guard is

```c++
if (strlen(aCommandName) > IGTL_COMMAND_NAME_SIZE) return 0;
strcpy((char*)m_CommandName, aCommandName);
```

`m_CommandName` is `igtlUint8[128]`. When the input is exactly 128
characters long, `strlen` returns 128 which does NOT exceed 128, so
the guard passes; `strcpy` then writes 129 bytes (128 chars + the
terminating null) starting at `m_CommandName[0]`, writing the null
at `m_CommandName[128]` — one byte past the buffer. Classic strcpy
off-by-one. The adjacent field in memory is `m_Encoding`
(igtlUint16), so the single-byte overwrite corrupts the low byte of
the encoding value unless compiler-inserted padding absorbs it.

**Required conformant behavior:** bound-check with `>=` (preserving
room for a null) OR skip the null terminator entirely by using a
counted copy (`memcpy(..., N)` with N = min(strlen, 128)) and padding
the remainder with zeros. The fixed-width field on the wire is 128
bytes null-padded; there is no need for a C string terminator inside
`m_CommandName` at all.

**Maps to audit commit:** not yet addressed upstream. Flagged for
the NG reimplementation.

## Messages formerly deferred — all resolved

Every data-carrying message type in the upstream protocol now has a
schema under `spec/schemas/`. The following entries were originally
on a "deferred pending meta-schema extensions" list but turned out
to be expressible with the existing vocabulary of flat fields,
sibling-field counts, struct elements, and `count_from: remaining`:

- **BIND** — structured header (type_id + body_size per child) +
  name table (byte-counted raw) + bodies (remaining). The
  cross-referencing between header entries and bodies is receiver
  parsing logic, not wire shape.
- **POLYDATA** — 40-byte fixed header with per-section byte counts +
  struct(x,y,z) point array + byte-counted topology sections +
  struct attribute headers + remaining attribute data.
- **POSITION** — `float32[3]` + `float32[] count_from: remaining`;
  body-size variants (12/24/28) map to 0/3/4 quaternion elements.
- **NDARRAY** — `uint8 + uint8 dim + uint16[dim] + uint8[] remaining`.
- **COLORTABLE** — `int8 + int8 + uint8[] remaining`.
- **COMMAND** — sibling-field count (`count: "length"`).
- **IMAGE** — 72-byte flat header + `uint8[] remaining`.
- **VIDEO** — 76-byte flat header + `uint8[] remaining`.

Note: there is no IMAGE2 message type. `Testing/img/igtlTestImage2.raw`
is a test image payload, not a distinct wire format.
