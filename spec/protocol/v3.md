# OpenIGTLink Protocol — v1/v2/v3

**Status:** Draft skeleton. Normative content to be filled in by extracting
observable behavior from the hardened reference C++ implementation. Until
this document is complete, the **existing library's wire behavior is the
authoritative spec**, captured in the `../corpus/` test vectors.

**Audience:** implementers of OpenIGTLink-compatible software, auditors
verifying SOUP compliance, and developers of new language ports.

## Overview

OpenIGTLink is a binary, length-prefixed message protocol for real-time
communication in image-guided therapy and surgical navigation. Messages
are exchanged over TCP, UDP, or WebSocket. Every message consists of:

1. A fixed-size **header** (58 bytes).
2. In v2 and v3, an optional **extended header** of variable length.
3. A **body** whose layout is determined by the message type.
4. In v2 and v3, an optional **metadata block** at the end of the body.

All multi-byte integers and floats on the wire are **big-endian**.

## Protocol versions

The protocol's `version` field in the header (`uint16`) takes values:

- `1` — original format. Header + body only. No metadata, no extended header.
- `2` — adds metadata block at the end of the body.
- `3` — adds the extended header; metadata block position adjusted.

Peers negotiate version implicitly: a receiver accepts any supported
version the sender emits. A `CAPABILITY` message exchange enumerates
supported message types.

**Legacy note:** the v2→v3 transition repositioned the metadata block.
v3 places metadata at the body *end*, after the content, not after the
extended header. This caused several independent implementations to
misplace it; the corpus pins the correct offset.

## Header

58 bytes, fixed layout. All fields big-endian unless noted.

| Offset | Size | Type      | Field         | Notes |
|-------:|-----:|-----------|---------------|-------|
|     0  |    2 | uint16    | version       | 1, 2, or 3 |
|     2  |   12 | char[12]  | type          | ASCII, null-padded message type string |
|    14  |   20 | char[20]  | device_name   | ASCII, null-padded |
|    34  |    8 | uint64    | timestamp     | seconds (upper 32 bits) + fraction (lower 32 bits) since Unix epoch |
|    42  |    8 | uint64    | body_size     | bytes in body (includes extended header and metadata in v2/v3) |
|    50  |    8 | uint64    | crc           | CRC-64 of the body bytes |

**Legacy note:** the `timestamp` is split into two `uint32`s (seconds
and fraction-of-second) on the wire but is often treated as a single
`uint64` in implementations. The fraction field uses the high bit of
the low word for the sign — this should not be relied on.

### CRC

CRC-64 of the entire body (including extended header and metadata
where present). The polynomial is ECMA-182 (the same variant used by
xz/lzma). Implementations MUST verify CRC before acting on any body
content. A CRC mismatch is a hard rejection; implementations MUST NOT
attempt partial parsing.

## Extended header (v3 only)

Variable-length, located at the start of the body in v3 messages.

| Offset | Size | Type   | Field                 | Notes |
|-------:|-----:|--------|-----------------------|-------|
|     0  |    2 | uint16 | ext_header_size       | bytes in this extended header |
|     2  |    2 | uint16 | metadata_header_size  | bytes in the metadata-index section |
|     4  |    4 | uint32 | metadata_size         | bytes in the metadata-body section |
|     8  |    4 | uint32 | message_id            | opaque sender-assigned identifier |
|   12+  |      | bytes  | (extension fields)    | reserved for future use |

The actual extended header size is `ext_header_size`, which may
exceed 12 bytes in future revisions. Implementations MUST skip any
bytes beyond the known fields up to `ext_header_size` without error.

## Metadata block (v2 and v3)

A metadata block carries arbitrary key/value pairs. Layout:

- **v2:** appended after content; located by subtracting content size from body_size.
- **v3:** located by the sizes declared in the extended header.

### Metadata index (header section)

- `uint16 count` — number of entries
- Per entry, in index order:
  - `uint16 key_size`
  - `uint16 value_encoding` — IANA MIBenum charset code (3 = US-ASCII; 106 = UTF-8)
  - `uint32 value_size`

### Metadata body

Sequential, packed, no padding:

- key bytes (`key_size` bytes, UTF-8, no terminator)
- value bytes (`value_size` bytes, encoded per `value_encoding`)

### Validation requirements

- Key size MUST be > 0.
- Total size of all entries MUST equal the declared `metadata_size`.
- Key and value size sums MUST NOT overflow the declared metadata region.
- Keys SHOULD be unique; behavior on duplicate keys is implementation-defined.

**Legacy note:** several audit findings in the existing library's
metadata parser stem from missing integer-overflow checks on these
size fields. A conformant implementation rejects metadata blocks whose
declared sizes do not fit within the body.

## Body

The body layout is fully determined by the `type` field. See the
machine-readable schemas under [`../schemas/`](../schemas/) for the
normative description of each message type. The prose description
below is a human-readable overview.

### Message type categories

- **Data messages** — TRANSFORM, POSITION, IMAGE, NDARRAY, POINT,
  TRAJECTORY, TDATA, QTDATA, STRING, SENSOR, POLYDATA, COLORTABLE,
  IMGMETA, LBMETA, VIDEOMETA, VIDEO, STATUS, CAPABILITY, BIND, UNIT,
  COMMAND.
- **Query messages** — `GET_*` (request a current snapshot of a data
  type), `STT_*` (request a stream), `STP_*` (stop a stream),
  `RTS_*` (reply-to-start / reply-to-stop status).

### Message type strings

Each type has a 12-byte ASCII string, null-padded. The canonical
string is specified in the schema for each message type. Type strings
are case-sensitive.

**Legacy note:** the 12-byte width has caused several historical
audit issues when type strings were memcpy'd without null
termination. Implementations MUST treat the field as fixed-width
bytes, not a C string, and MUST null-terminate before any use as a
C string.

### STATUS

Reports the operational status of a device or the outcome of a
previous request. See [`../schemas/status.json`](../schemas/status.json)
for the normative wire layout.

**Body layout** (variable-size, minimum 31 bytes):

| Offset | Size | Type      | Field          |
|-------:|-----:|-----------|----------------|
|     0  |    2 | uint16    | code           |
|     2  |    8 | int64     | subcode        |
|    10  |   20 | char[20]  | error_name     |
|    30  |  N+1 | char[N+1] | status_message |

`status_message` occupies all remaining body bytes. Its final byte
MUST be `0x00`; the string value is the preceding `N` bytes. An empty
status_message is represented as a single `0x00` byte (N=0), giving a
minimum body size of 31 bytes.

**Status codes** (defined values):

| Value | Name              | Meaning |
|------:|-------------------|---------|
|     0 | INVALID           | Sentinel, not a valid runtime status |
|     1 | OK                | Normal operation |
|     2 | UNKNOWN_ERROR     | Generic error |
|     3 | PANIC_MODE        | Emergency (upstream constant spelled `PANICK_MODE`) |
|     4 | NOT_FOUND         | File, configuration, or device not found |
|     5 | ACCESS_DENIED     | |
|     6 | BUSY              | |
|     7 | TIME_OUT          | Timeout or connection lost |
|     8 | OVERFLOW          | Overflow or cannot be reached |
|     9 | CHECKSUM_ERROR    | |
|    10 | CONFIG_ERROR      | |
|    11 | RESOURCE_ERROR    | Insufficient memory, storage, etc. |
|    12 | ILLEGAL_INSTRUCTION | Unknown or unsupported instruction |
|    13 | NOT_READY         | Device starting up |
|    14 | MANUAL_MODE       | Device does not accept commands |
|    15 | DISABLED          | |
|    16 | NOT_PRESENT       | |
|    17 | UNKNOWN_VERSION   | |
|    18 | HARDWARE_FAILURE  | |
|    19 | SHUT_DOWN         | Exiting or shutdown in progress |

Receivers MUST accept unknown code values without rejection, so the
enumeration can grow over time without breaking compatibility.

### STRING

Carries a character string with an explicit character-encoding hint.
See [`../schemas/string.json`](../schemas/string.json) for the normative
wire layout.

**Body layout** (variable-size, minimum 4 bytes):

| Offset | Size | Type    | Field    |
|-------:|-----:|---------|----------|
|     0  |    2 | uint16  | encoding |
|     2  |    2 | uint16  | length   |
|     4  |    N | byte[N] | value    |

`value` is `length` bytes of string data in the encoding specified by
the `encoding` field. There is no terminator byte. An empty string is
legal (`length=0`, body size 4).

The `encoding` field is an IANA MIBenum value. Common values:
- `3` — US-ASCII
- `106` — UTF-8

Receivers MUST accept unknown encoding values without rejection; if
the encoding is not recognized, the payload MAY be treated as opaque
bytes. The 16-bit `length` field caps the maximum string size at
65535 bytes.

### SENSOR

Reports a vector of scalar sensor readings with an attached SI unit
descriptor. See [`../schemas/sensor.json`](../schemas/sensor.json) for
the normative wire layout.

**Body layout** (variable-size; total = 10 + `larray` × 8 bytes):

| Offset | Size | Type        | Field   |
|-------:|-----:|-------------|---------|
|     0  |    1 | uint8       | larray  |
|     1  |    1 | uint8       | status  |
|     2  |    8 | uint64      | unit    |
|    10  |  N×8 | float64[N]  | data    |

Where `N = larray` (range 0..255). Maximum body size is therefore
10 + 255 × 8 = 2050 bytes.

The `unit` field is a bit-packed descriptor of an SI unit (prefix
code + up to six base- or derived-unit codes with signed exponents).
At the wire level it is a single big-endian `uint64`; decoding its
internal structure is optional and can be deferred to a helper.

The `status` byte has no standardized values. Receivers SHOULD ignore
unrecognized values and MUST NOT reject a message based on its
content.

### CAPABILITY

Advertises the list of OpenIGTLink message types a device accepts.
See [`../schemas/capability.json`](../schemas/capability.json) for the
normative wire layout.

**Body layout** (variable-size; always a multiple of 12 bytes):

| Offset | Size  | Type            | Field            |
|-------:|------:|-----------------|------------------|
|     0  | 12×N  | char[12]×N      | supported_types  |

Each element is an OpenIGTLink message-type identifier as a 12-byte
ASCII string, null-padded on the right. The number of entries is
implicit: `N = body_size / 12`.

A body whose size is not a multiple of 12 is malformed and MUST be
rejected. An empty list (`N = 0`, `body_size = 0`) is legal but
unusual.

**Typical usage:** sent in response to a `GET_CAPABIL` query during
session setup. Peers inspect the list to decide which message types
to send on the session.

## Transport

### TCP

The default transport. Messages are sent back-to-back with no framing
other than the header-declared `body_size`. Connection-oriented; no
retry semantics at the protocol level.

### UDP

Used for VIDEO and high-rate tracking streams. Messages that exceed
the path MTU are fragmented using an OpenIGTLink-specific RTP-lookalike
wrapper. The wrapper is not RFC 3550 compliant.

**Legacy note:** the current UDP/RTP wrapper invents its own fragment
markers (`0x8000` begin, `0xE000` end, `0x0000` no-fragment). It is
NOT interoperable with standard RTP implementations. NG (v4) replaces
this with RFC-compliant RTP payload formats; v3 peers continue to use
the legacy wrapper for backward compatibility.

### WebSocket

Each OpenIGTLink message is sent as a single binary WebSocket frame.

### TLS

Not specified by v3. Implementations MAY wrap any transport in TLS
externally. NG defines explicit TLS negotiation and requirements.

## Conformance

An implementation is conformant iff it passes every entry in
[`../corpus/`](../corpus/):

- Decoder test: given `<entry>.bin`, produce a semantic result matching
  `<entry>.json`.
- Encoder test: given `<entry>.json`, produce bytes exactly matching
  `<entry>.bin`.
- Negative test: given an entry under `corpus/negative/`, reject it
  with an error matching the entry's declared expected behavior.

In-library tests are insufficient to establish conformance; the corpus
is the sole source of truth for wire compatibility.

## Known upstream correctness issues

Behavior observed in the reference C++ implementation at pinned SHA
`94244fe` that differs from what a correct implementation should do.
Each item below identifies a bug to be preserved as a *negative-corpus*
expectation (i.e., the corpus will pin the *correct* rejection
behavior, not the buggy upstream behavior). When the hardening commits
from the patched library land upstream, these entries should be
revisited and either resolved or retained as documentation of the
fixed defect.

### U-1: `SetErrorName` strncpy can leave `error_name` non-null-terminated

**Location:** `Source/igtlStatusMessage.cxx:72-77`, `UnpackContent` at the same file `:132-133`.

**Behavior:** The code writes `m_ErrorName[19] = '\0'` and then calls
`strncpy(m_ErrorName, name, 20)`. If `name` is 20 or more bytes,
strncpy writes 20 characters and overwrites the pre-placed null.
Subsequent use of `m_ErrorName` as a C string OOB-reads past the
buffer.

**Required conformant behavior:** either reserve byte 19 as a hard
null (effective 19-character limit) or treat the field as fixed 20
bytes with no null-termination assumption. Receivers MUST defensively
null-terminate before C-string use.

**Maps to audit commit:** upstream hardening branch commit `aade268`
("null termination, inet_addr, type safety").

### U-2: `StatusMessage::UnpackContent` reads before `m_StatusMessage` on short body

**Location:** `Source/igtlStatusMessage.cxx:137`.

**Behavior:** The null-terminator check is
`m_StatusMessage[CalculateReceiveContentSize(isUnpacked) - IGTL_STATUS_HEADER_SIZE - 1]`.
If the attacker sets `body_size` such that the content size is ≤ 30,
this indexes with a negative offset and reads from memory before the
`m_StatusMessage` buffer.

**Required conformant behavior:** reject any STATUS message with
`body_size < 31` before any body access. The minimum body is the
30-byte fixed header plus the single mandatory null terminator byte.

### U-3: `StatusMessage::UnpackContent` casts `m_Content` without size check

**Location:** `Source/igtlStatusMessage.cxx:127-130`.

**Behavior:** The function casts `m_Content` to `igtl_status_header*`
and reads 30 bytes without first verifying `body_size >= 30`. A
malformed short body results in OOB reads or potential null deref.

**Required conformant behavior:** validate body size against the
fixed-header size before any field access.

### U-4: UDP/RTP wrapper fragment markers are not RFC 3550

**Location:** `Source/igtlMessageRTPWrapper.cxx`.

**Behavior:** The UDP fragmentation scheme invents its own markers
(`0x8000` begin, `0xE000` end, `0x0000` no-fragment) rather than using
RFC 6184 / 7798 / 7741 RTP payload formats. Not a correctness bug per
se, but an interop limitation that has also produced a heap overflow
in the fragment reassembler.

**Required conformant behavior:** v3 peers preserve the legacy wrapper
format for wire compatibility. NG (v4) replaces it with RFC-compliant
RTP. See the "Transport: UDP" section above.

### U-5: `StringMessage::UnpackContent` OOB read on attacker-controlled length

**Location:** `Source/igtlStringMessage.cxx:135`.

**Behavior:** The unpacker calls
`m_String.append(string, string_header->length)` without verifying
`length` against the actual body size available. An attacker sending
`body_size=4` with `length=65535` causes a 65535-byte OOB read starting
at offset 4 of the message content buffer.

**Required conformant behavior:** reject any STRING message where
`body_size < 4 + length`. The check MUST occur before any read of the
string payload.

**Maps to audit commit:** upstream hardening branch — STRING is
covered by the general `CalculateReceiveContentSize(isUnpacked)`
pattern introduced in commits `f3640aa`, `5c0499e`, and `4ad0923`
("Fix StringMessage validation order").

### U-6: `SensorMessage::UnpackContent` OOB on attacker-controlled `larray`

**Location:** `Source/igtlSensorMessage.cxx:168`; underlying byte-swap
loop at `Source/igtlutil/igtl_sensor.c:44-47`.

**Behavior:** `UnpackContent` calls `igtl_sensor_convert_byte_order`
before verifying the body is large enough. That function iterates
`i < larray` and byte-swaps `tmp[i]` (8 bytes each) in place. An
attacker sending `body_size=10` (fixed header only) with `larray=255`
causes a 2040-byte OOB read and 2040-byte OOB write immediately past
the receive buffer. The subsequent `data[i]` reads at lines 176–179
amplify the OOB read.

**Required conformant behavior:** reject any SENSOR message where
`body_size < 10 + larray * 8`. Check MUST occur before any data-array
access, including the byte-swap pass.

**Maps to audit commit:** covered by the general
`CalculateReceiveContentSize(isUnpacked)` pattern used across the
hardening branch; SENSOR-specific coverage is part of the
`0ad4b4d` group ("Reject malformed element-based messages with
partial elements").

## Messages deferred pending meta-schema extensions

The following upstream message types have wire layouts that the
current meta-schema cannot yet express cleanly. They are documented
here so the path forward is explicit.

- **POSITION**, **QTRANS-using messages** — body is a size-discriminated
  variant: 12 bytes (position only), 24 bytes (position + 3-element
  compressed quaternion), or 28 bytes (position + full quaternion).
  Requires meta-schema support for body-size-discriminated unions.

- **POINT**, **TRAJECTORY**, **TDATA**, **QTDATA**, **IMGMETA**,
  **LBMETA** — each element is a per-type struct (name + metadata +
  payload per entry); element count is implicit in the body size.
  The `count_from: remaining` plus inline-element-descriptor
  mechanisms from CAPABILITY cover the count side; the remaining
  need is **reusable struct definitions** (today's inline
  ElementDescriptor is single-purpose; a full struct layout with
  multiple named fields is not yet expressible).

- **COLORTABLE** — element count is derived from a lookup of two
  enum fields (`indexType`, `mapType`) into a small table of
  `(N, S)` pairs, not from body_size alone. Requires either a
  lookup-driven count mechanism or explicit description of the
  size-derivation function.

- **NDARRAY** — variable rank and per-dimension sizes, with data count
  = product of dimension sizes. Requires the same `count: remaining`
  mechanism plus array-of-product support.

- **IMAGE**, **IMAGE2** — extended header with coordinate-transform
  matrix, subvolume offset/size, pixel array. Shape is expressible
  but large; deferred for tactical scoping.

- **COMMAND**, **BIND**, **POLYDATA** — complex multi-section layouts
  that warrant dedicated schemas and additional meta-schema field
  types.
